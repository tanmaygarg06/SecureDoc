# Secure Cloud Document Management System (AWS S3 + IAM)

A full-stack app where users sign up, sign in, and upload/download/delete
files and folders that are stored directly in an **S3 bucket**. Access to
that bucket is controlled entirely through **IAM** — no other AWS service
is used. There is no separate database: user accounts are stored as JSON
objects in S3 as well.


**Why it satisfies "S3 + IAM only":**
- **S3** stores everything: user records, files, and folders (folders are just
  keys ending in `/`, S3's standard convention).
- **IAM** is the only access-control layer: a dedicated IAM user has a policy
  that allows it to read/write/list/delete objects in exactly one bucket, and
  nothing else. The Express backend uses those credentials; the browser never
  talks to AWS directly except through short-lived **presigned URLs** (also an
  IAM-governed S3 feature) for viewing/downloading files.

---

## 🚀 Features
- **User authentication** with JWT
- **Admin dashboard** to view all users and their files
- **File management**: upload, download, delete, create folders
- **Presigned URLs** for secure file access
- **S3 + IAM only** (no database, no other AWS services)

## 2. AWS setup

### Step 2.1 — Create the S3 bucket

1. AWS Console → **S3** → **Create bucket**.
2. Bucket name: something globally unique, e.g. `bucket_name`.
3. Region: pick one close to you (e.g. `ap-south-1` for India) — remember it,
   you'll need it in `.env`.
4. **Block all public access**: keep this **ON** (checked). We never make the
   bucket public — files are only ever served through short-lived presigned
   URLs generated by the backend.
5. Leave everything else default → **Create bucket**.

### Step 2.2 — Configure CORS on the bucket

The browser will `GET`/`PUT` presigned URLs directly against S3 for
upload/download in some flows, and the app itself calls S3 from the backend —
but to be safe (and to support any future direct-from-browser uploads), add a
CORS configuration:

Bucket → **Permissions** tab → **Cross-origin resource sharing (CORS)** → paste:

```json
[
  {
    "AllowedHeaders": ["*"],
    "AllowedMethods": ["GET", "PUT", "POST", "DELETE", "HEAD"],
    "AllowedOrigins": ["http://localhost:8080"],
    "ExposeHeaders": ["ETag"],
    "MaxAgeSeconds": 3000
  }
]
```

Replace/add the origins with wherever your frontend is actually hosted.

### Step 2.3 — Create the IAM user (or role) for the backend

1. AWS Console → **IAM** → **Users** → **Create user**.
2. Name: `name`.
3. Do **not** grant console access — this user only needs **programmatic
   access** (access keys) for the backend server.
4. Skip attaching a managed policy for now — you'll attach a custom
   least-privilege policy in the next step.

### Step 2.4 — Attach a least-privilege IAM policy

IAM → **Policies** → **Create policy** → **JSON** tab → paste (replace
`YOUR_BUCKET_NAME`):

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ListBucketOnly",
      "Effect": "Allow",
      "Action": ["s3:ListBucket"],
      "Resource": "arn:aws:s3:::YOUR_BUCKET_NAME"
    },
    {
      "Sid": "ObjectReadWriteDelete",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject",
        "s3:DeleteObject"
      ],
      "Resource": "arn:aws:s3:::YOUR_BUCKET_NAME/*"
    }
  ]
}
```

Name it `BucketPolicy`, create it, then go back to the
`name` user → **Add permissions** → **Attach policies directly**
→ select `BucketPolicy`.

This means: even if the backend server were compromised, the exposed AWS
credentials could only ever touch this one bucket — nothing else in the AWS
account.

### Step 2.5 — Generate access keys

IAM user `name` → **Security credentials** tab → **Create
access key** → choose **Application running outside AWS** → create it.
Copy the **Access key ID** and **Secret access key** — you'll paste these
into `.env` in the next section (you won't be able to see the secret again).

## 3. Backend setup (local development)

```bash
git clone https://github.com/your-username/secure-cloud-docs.git
cd secure-cloud-docs/backend
npm install
```

Copy `.env.example` → `.env` and fill in your details:

```
PORT=8080
AWS_REGION=ap-south-1                     # region you created the bucket in
AWS_ACCESS_KEY_ID=AKIA...                 # from step 2.5
AWS_SECRET_ACCESS_KEY=xxxxxxxxxxxxxxxx    # from step 2.5
S3_BUCKET_NAME=securedocs-yourname-2026   # your bucket name
JWT_SECRET=some_long_random_string        # e.g. output of: openssl rand -hex 32
JWT_EXPIRES_IN=7d
ADMIN_EMAILS=admin@example.com            # any account signing up with this email becomes admin
```

Run it:

```bash
npm start
# backend running on http://localhost:8080
```

The backend also serves the `frontend/` folder as static files, so once it's
running you can simply open **http://localhost:8080** in your browser — no
separate frontend server needed.

## 4. Using the app

1. Open `http://localhost:8080`.
2. Click **Create one** → sign up. If you use the email listed in
   `ADMIN_EMAILS`, that account becomes an **admin** automatically; every
   other email becomes a normal **user**.
3. As a **user**: on the dashboard you can
   - **Upload File** — uploads directly into `users/<yourId>/<currentFolder>/`
     in S3.
   - **New Folder** — creates an empty S3 object key ending in `/`.
   - Click a folder name to navigate into it (breadcrumbs let you go back).
   - **View** — opens the file in a new tab via a presigned URL (5-minute
     expiry).
   - **Download** — same, but forces a download instead of inline view.
   - **Delete** — deletes a file, or a folder and everything inside it
     (recursively removes every S3 object under that prefix).
4. As an **admin**: `admin.html` lists every registered user (pulled from
   `_system/users/*.json` in S3). Click **View Files** next to any user to
   browse (read-only) everything they've uploaded.
5. **Profile** page shows your name, email, role, and join date, with a
   **Sign Out** button. Signing out just clears the JWT from local storage
   client-side.

## 5. How each feature maps to S3 + IAM

| Feature | How it works |
|---|---|
| Sign up / Sign in | Password hashed with bcrypt, user record written/read as a JSON object at `_system/users/<email>.json` in S3. A JWT is issued for session auth (no AWS service needed for sessions). |
| Admin — see all users | `ListObjectsV2` with prefix `_system/users/`, then `GetObject` on each. |
| Upload file/folder | `PutObjectCommand` to `users/<userId>/<path>/<filename>`. A folder is `PutObjectCommand` with an empty body and a key ending in `/`. |
| Delete file/folder | `DeleteObjectCommand` for a single file; for a folder, `ListObjectsV2` everything under that prefix then `DeleteObjectsCommand` in batch. |
| View / Download | `GetObjectCommand` wrapped in `getSignedUrl` (presigned URL, 5 min expiry) — the browser hits S3 directly for the actual bytes, but only ever with a URL the backend generated after checking the user owns that key. |
| Isolation between users | Every user's objects live under their own `users/<userId>/` prefix; the backend rejects any key that doesn't start with the caller's own prefix (except for admins browsing via the dedicated admin route). |
| Security boundary | Enforced by the IAM policy attached to the backend's IAM user (scoped to one bucket) + JWT auth + per-request ownership checks in `fileService.js`. |

## 6. Troubleshooting

- **`AccessDenied` errors from S3** → double-check the IAM policy's
  `Resource` ARNs match your exact bucket name, and that the policy is
  attached to the user whose keys are in `.env`.
- **CORS errors in the browser console** → confirm the bucket CORS config
  includes the exact origin (protocol + host + port) you're loading the
  frontend from.
- **"Invalid email or password" right after signing up** → email matching is
  case-insensitive and trimmed automatically; make sure you're not adding
  extra spaces.
- **Presigned URL expired** → they last 5 minutes by design; just click
  View/Download again to generate a fresh one.
